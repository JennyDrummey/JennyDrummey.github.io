Organization of the InstallManuals Flare Project

The project is organized by manual. Several sections are used across multiple manuals, with conditional tags applied.

The Version History topic (Appendix A for the Installation Manual and the SRHS Guide) is updated for each manual with all updates made to the manual in the release. Appendix A is not delivered with the final system, it is used so that the documentation team can track changes for each version. The most recent entry is at the top.

Update the Installation Manuals

The following steps describe the collaboration, review, approval and final formatting process to get the Installation Manuals ready to publish for each software release.

Members of the Documentation Team take turns updating the Manuals for each release.

Follow these steps to update the install manuals:

1. The Tech Writer synchronizes their local master branch with the BackOfficeAssoc dsp-install-manuals master branch. NOTE: If you do not have a local copy of the install manuals on your laptop, see Clone the dsp-install-manuals repo.

2. The TW creates a new Staging branch based on the Master branch in the dsp-install-manuals repo.

3. The TW creates a release folder on SharePoint in the Published folder for the current release, creates new Word docs for review with the updated version number in the file titles. NOTE: Ensure Track Changes is turned on for this version so that reviewers can find updates easily. The TW will include a link to this folder in the next step when they create the Jira ticket.

4. The TW creates a Jira ticket in the DSP70 project for updating the manuals for this release.

5. The TW links all tickets that describe updates to manuals to this ticket to track the updates.

6. The TW and contributors review and update the manuals in Sharepoint in the Published folder and Flare as needed. If major changes have been made to the Installation Manuals, Support will test these steps.

7. Once the collaboration is complete and the content is final, the approver sets the Documentation Status field to of the installation manual Jira issue to Approved.

8. The TW makes the changes in the Stangingnnn branch of the Flare project. Refer to Update Online Help for more information. After all changes are made, the TW builds the project in the dev environment in Jenkins.

9. The TW sets the Documentation Status field of the installation manual Jira issue to Peer Review and requests a peer review from another tech writer on the team in the doc_checkouts Slack channel.

10. After peer review is complete, if needed,the Author updates the Falre project and builds the install manuals Flare project in dev/prod.

11. The TW sends notification that the final manuals are ready to Support with a link to the production version and then closes the Jira ticket.

IMPORTANT: The master branch must be updated with the latest changes at the close of each release, and a new Staging branch must be created at the beginning of every release cycle from the master branch by the Tech Writer who is handling the manuals that release cycle. Sometimes customers need manuals specific to the version of SST that they are running, so the Staging branches must not be updated or deleted once they have been merged into the master branch.

Clone the dsp-install-manuals repo

To clone the dsp-install-manuals repo, follow the instructions here, but all Staging branches should be based on the Master branch from this repo.

Synchronize your master branch with the BackOfficeAssoc GitHub Repo

WARNING: You MUST perform these steps each time you checkout a branch to ensure you are working in the latest copy of the master branch.

  1. In Windows-Explorer, right-click on your dsp-install-manuals local folder and select Git Bash Here.

NOTE: If you do not have a local dsp-install-manuals repo, see Clone the dsp-install-manuals repo for instructions.

  1. Switch to the master branch.
    Note: The git command to switch to the master branch is git checkout master.
  2. Enter the git command, git pull origin master TIP: If at this point you wish to see a list of branches that are available in your local repo, enter the command git branch. All of the available local branches are listed.

Create a branch for your updates based on the master branch

  1. Navigate to the master branch in Git Bash if you are not already in it. IMPORTANT: Make sure you synchronize your Master branch before completing step 2.
  2. Enter the command git checkout -b <new branch name> NOTE: Name the branch after the release number. Use the naming convention StagingNNN (NNN is the release number).
  3. Open the project in Flare and update the DSPVersion variable to the current release version.
  4. Enter the command git add ..
  5. Enter the command git commit -m " <commit message>".
  6. Enter the command git push origin <new branch name>.

Create a Release Folder on SharePoint

  1. Navigate to the Product Documentation > Published folder.
  2. Click +New and select Folder.
  3. Enter the release number into the Folder Name field.
  4. Click Create.
  5. Open the prior version folder and select the Word files of the different manuals.
  6. Click Copy to and select the new release folder you just created.
  7. Update the titles of the manuals to reflect the new release number.
  8. Open the new manuals and accept all changes made in the previous versions. NOTE: You caould also create a new version of the Word document for any of the guides in Flare. Set the correct TOC for hte print version (the one in the archive) folder and the Word targets will still work).
  9. Ensure Track Changes is on in the Word document to record updates.

Create a JIRA Ticket for the Manuals

  1. In JIRA click the plus sign on the left side of the System Dashboard.
  2. Select the DSP70 project from the Project dropdown.
  3. Select Epic from the Issue Type dropdown.
  4. Enter an epic name in the Epic Name field. E.g. "Documentation for 7.0.1 Release (Manuals and Release Notes)".
  5. Enter a summary in the Summary field. E.g. “Update manuals with changes for 7.0.1.”
  6. Enter a description in the Description text box. E.g. “Word document links to the manuals with 7.0 updates. Link to Sharepoint folder.” NOTE: Make sure to include a link to the SharePoint folder that you created for this release.
  7. Add the Fix Version, as the release number.
  8. Add the Assignee, as yourself.
  9. Click Update.

Link Tickets Related to Manuals to JIRA Epic

Add links to the epic to any tickets that affect the installation manual.
