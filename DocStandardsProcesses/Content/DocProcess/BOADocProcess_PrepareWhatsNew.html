Prepare SST What's New Topic for Release

When all content is finalized for a release (as in PM has approved all content and it has been added to the Flare project by the Technical Writers and peer reviewed), a Technical Writer reviews the entire What’s New topic before the SST is released.

NOTE: The SKT has a What's New topic that is not released in a pdf version, and is updated according to deployments in the cloud.

The What’s New topic is often the first one the user sees and must be free of grammar issues, typos, and broken links. This help topic is used as the basis for the release notes, a secured PDF that is also distributed with each SST release. Refer to Prepare Release Notes for more information.

The single, version-specific What’s New topic is formatted with condition tags in Flare. Two versions of the What’s New topic are created from this topic: one for the Syniti Solutions version of the help and one for the SAP Accelerators version.

Update Release Date

The official release date is listed at the top of the What’s New topic. Update this with the release date, which also matches the release date in the Version History of the installation manuals.

Review Structure of Release Notes

Content is organized into these sections.

  • New Features
  • Enhancements
  • Resolved Issues
  • Known Issues
  • Enhancement Requests from Users

These headings display as links at the start of the What’s New topic. Not every release will have content in each of these sections. Remove (or add, as needed) these links at the start of the What’s New topic, depending on that version’s contents.

Each section (New Features, Enhancements, etc.) is divided into subsections based on product name and component. Review the headings and structure in the What’s New topic, confirming that condition tags are applied correctly as noted in the next sections.

Review Content of New Features Section

This section describes major features that are introduced with the version such as a new WebApp, or a feature in an existing WebApp that has not been available before. These may be described in JIRA with an issue type of Story or Epic.

If a component has multiple new features, use a bulleted list. If there is only one new feature, do not use a bulleted list.

Each new feature contains a link to the help topic(s) that describe it.

Review Content of Enhancements Section

This section contains minor updates (often described in a JIRA issue type of Story).

If a component has multiple enhancements, use a bulleted list. If there is only one enhancement, do not use a bulleted list.

Each new enhancement contains a link to the help topic(s) that describe it.

Review Resolved Issues Section

This section describes resolutions to customer issues.

NOTE: QE Bugs that were introduced with the current (not yet released) version are not included in the Resolved Issues section. For example, if QE discovers a bug with a new feature that is being released with the current version, do not include that bug in the release notes.

Resolved issues are described in JIRA issues that match one or more of the following:

  • Were entered by the support team
  • Contain links to a Zendesk customer ticket
  • Mention a customer in the comments
  • Were entered by PM as an internally reported Bug.

Each issue description contains this information:

  • The nature of the issue, in other words, what was broken.
  • What was done to fix it, if the fix is visible to the user. Back end updates, such as those related to stored procedures, tables, or views are not included.
  • How the product functions with the fix
  • At the end of the description add a reference to the JIRA issue number enclosed in brackets for example, [DSP-1234]

Enhancement Requests from Users

Enhancement Requests from Users are described in JIRA issues that:

- Have the label DSP Enhancement Request

- Have the label IdeasPortal

- Are the result of online help feedback

Each Enhancement Request contains a link to the help topic(s) that describe it.
