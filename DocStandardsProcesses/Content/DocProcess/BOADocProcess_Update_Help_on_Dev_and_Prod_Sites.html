Update Help on Dev and Prod Sites

The dev site is for Tech Writers to test that the online help works. The prod site is the site that users access directly from the product when they click the ? on any page.

The process for updating help is the same for the product help (in the dps-docs repo) and the installation manuals/Requiements/SAP Config guides (stored in the dsp-install-manuals).

Each SST release has a branch in the dsp-docs repo with the naming convention develop_nnn, where nnn is the release number (develop_701, for example). After a pull request is merged to the branch develop_nnn, a Technical Writer can then log in to Jenkins and build the help on the dev site, prod site, or both.

Each SST release has a branch in the dsp-install-manuals repo with the naming convention Stagingnnn, where nnn is the release number (Stagini701, for example). After a pull request is merged to the branch develop_nnn fo Stagingnnn, a Technical Writer can then log in to Jenkins and build the help on the dev site, prod site, or both.

When jenkins builds help for these versions, it copies the output to a site accessed by a specific URL.

Tip: You must Build and Publish Help Targets in Flare before building in Jenkins for the SKT

Update the Help on the Dev and Prod Sites

To update the help on the sites for SST and the Syniti Guides:

  1. Log in to Jenkins.

  2. In the Name column, click DSPDocs.

  3. Click the Schedule a Build with parameters for [Target Name] icon for either

    • DSPDocDev

    • DSPDocProd.

    • DSPDocDev-Installs

    • DSPDocProd-Installs

Note: If you are updating a version of the help that has already been released, build the help in dev then in prod. If you are updating a version of the help that has not yet been released, build in dev as needed.

  1. Enter the version number in the VERSION field.

  2. Click Build. The build status displays in the left column under Build History.

  3. Open an incognito window in Chrome to view the help to confirm your updates are on the site.

To update the sites for the SKT you do not use Jenkins. Instead, you push content to two different repositories:

  1. Merge your branch to the develop branch. This runs a job to automatically update the develop site.

  2. After checking the develop site, merge the develop branch to the master branch. This pushes your updates to the production site.
