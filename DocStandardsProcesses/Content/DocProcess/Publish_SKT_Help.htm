Publish SKT Help

SKT Help updates can be published anytime and do not require a jenkins build. When a ticket on the Kyber board reaches the Prod swimlane, it is deployed and the related documentation should be published.

To publish Help:

  1. Prepare the help content as described in the Documentation Process Overview. Only content approved by a PM or Dev should be added to the help.

  2. Inform the Doc Team in the doc_checkouts Slack channel that you are going to check out the Help.

  3. Synchronize your local repo develop branch with the remote develop branch and create a new branch.

  4. Make updates and note them in your Author Notes.

    NOTE: If you are making changes to the What's New topic and they are the first changes made in a month, you must add that month to the drop-down list that displays on the What's New page.

  5. Build and Publish the target.

  6. Commit updates locally and push to remote.

  7. Merge your changes to the develop branch. This will automatically start a jenkins job that will copy the updated help system to the dev environment.

  8. When the changes are available on the develop site, ask for a peer review.

  9. When the peer review is complete, merge the develop branch to the master branch to push the changes to the prod environment.

Synchronize your local repo develop branch with the remote develop branch and create a new branch

  1. Right-click your local repo folder and select Git Bash Here from the dropdown menu to open the command prompt at that location.

  2. Review the current branch, which appears in parenthesis at the end of the line. If you are not currently in the develop branch, switch to the develop branch by entering the git command git checkout develop.

  3. Enter the git command git pull origin develop to synchronize your local copy of the develop branch with the remote copy.

    NOTE: You MUST perform these steps before you make updates to ensure you are working in the latest copy of the develop branch.

  4. Enter git command git checkout -b BranchName to create a personal branch in which to make edits, where BranchName is a brief description of the branch.

  1. Open the SKT help project in Flare and make updates.

Add Month to the Drop-down List on the What's New Page

If you are adding content to the What's New topic at the start of a new month, you must update the month option in the menu.

To add a new month to the drop-down list:

  1. In the Content Explorer, copy the SCReleases.htm file and rename it SCReleasesmmdd.htm.

  2. Open the WhatsNew master page, and locate the code for the menu.

  1. Add the menu option to the code by copying and pasting an option and updating it as needed.

  2. Click Save.

Build and Publish Target

To build and publish a target in Flare:

  1. Open Flare, click the down arrow on the Build Primary button and select HTML5 SKT Help.

  2. Open Project Organizer > Targets.

  3. Double-click HTML5 SKT Help (Primary) .

  4. Click Publishing tab.

  5. Select the Destination being updated: HTML5 SKT.

  6. Click the Edit Destination button; a new tab opens.

  7. Click the … button to edit the Directory.

  8. Select the destination folder: IGCHelp.

    NOTE: You must select your own IGCHelp output folder. For example, ~/CarolineStJohn/IGCHelp...

  9. Click Save.

  10. Click OK to the warning message that displays and close the tab.

  11. Click the Publish button on the topic toolbar of the Target Editor tab.

Proceed with committing changes in GitHub (git add., git commit, give push origin and create a pull request).

Commit updates locally and push to remote

  1. Return to Git Bash or open a new session.

  2. Navigate to your repo, and make sure you are in the correct branch. It is in parenthesis after your prompt.

  3. Enter the git command git status to see all of the changes you have made.

    NOTE: All of the updated files’ paths will appear in red.

    NOTE: If you do not see them, you may have forgotten to save or you are in the wrong branch.

  4. Enter the git command git add . to stage your updates for commit.

    NOTE: The file paths change from red to green after you stage them. Include the space and the period.

  5. Enter the git the command git commit -m "commit message" to commit staged updates.

    NOTE: Do not use special characters, like apostrophes, in message.

  6. Enter the git command git push origin BranchName to push your updates to the remote repo.

  7. Create pull request to merge BranchName into develop.

  8. Post to the doc_checkouts channel in Slack requesting a peer review.
